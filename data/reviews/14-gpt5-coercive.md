# Review: AWS IoT Core lottery fan-out POC

Baseline validation: `python -m compileall scripts` passed. I found no automated tests, no CI, no load test, and no deployment validation beyond manual steps.

I reviewed every shipped runtime/config/documentation file in the repo root (`scripts/`, `infra/`, `requirements.txt`, `.env.example`, `README.md`, `FINDINGS.md`) and cross-checked `FINDINGS.md` against `.planning/REQUIREMENTS.md`.

## FINDINGS.md verification summary

- **Unsupported / false-positive claims:** `DLVR-03`, `LOGS-03`, `CRED-01`, “All 17 requirements verified”, and the overall “Conditional Go” recommendation.
- **Partially supported claims:** QoS 1 redelivery behavior and persistent-session queuing are directionally supported for a tiny manual test, but not for 5,000 terminals and not as proof of terminal receipt.
- **Supported claim:** The document’s core conclusion that `Publish-Out status=Success` is broker dispatch, not receipt proof, is consistent with both the code and the rest of the write-up.

## Bottom line

This is still a POC, not a production-ready design for 5,000 lottery terminals. The two biggest blockers are:

1. **Security:** anonymous publish/subscribe access plus terminal/publisher impersonation.
2. **Correctness/observability:** no end-to-end delivery-proof mechanism after the project already proved CloudWatch logs are insufficient.

Until those are fixed, this should not be promoted past tightly controlled experimentation.
